NoSQL Databases

2015-05-08

As someone who's worked a lot with relational databases, I've always wondered how NoSQL Databases worked. So for this blog post, I've chosen to do a little research and find out what they're all about.

NoSQL databases started when there was a rise in the amount of data being stored by applications. Traditional SQL databases weren't made to handle the amount of data that needed to be stored. NoSQL databases were created to address this problem. Some of these advantages include:

Storing Unstructured Data

With relational databases, you have to create a schema ahead of time. Let's say you wanted to store employee records in a table. You'd then need to figure out everything you need to know about employees, such as name, address, salary, start date, etc. Which is fine, until you need to add something that you forgot, or maybe a new feature was implemented in your application, and it needs additional employee information. You now need to update the schema before you can start inserting this new data. NoSQL databases on the other hand, were made to allow inserts without a predetermined schema. This makes it easy to start inserting new types of information, in your database.

Auto Sharding

When creating a relational database, you need to store the entire database on a single server, to ensure that you can reliably access all of your data. As the server gets larger, you need to add more space to keep up with its size. This will eventually slow down query speed significantly. Alternatively, you can split the database into pieces, commonly referred to as shards. Each shard will have a portion of the database, and will sit on a separate server. You need to create software to manage and query multiple shards, which puts extra burden on the developers. NoSQL databases however, usually support auto-sharding. This means the database takes care of organizing and accessing the separate shards. From the developer's point of view, it looks like a single database. It makes things easier for the developer, and allows them to be less reliant on Database Administrators.

Conclusion

NoSQL databases have a lot of great advantages, but there are also some important disadvantages. NoSQL is still fairly new, so its use isn't as widespread as relational databases. This also means support and workers with skills in this area, will be harder to come by. Since NoSQL databases are generally open source projects, there generally isn't much standardization between the different implementations. Personally, I'd love to try using a NoSQL database. I'm sure it will also become a necessary skill, as they become more widely adopted.
